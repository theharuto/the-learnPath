# ✅ Mockito with Spring Boot — COMPLETE GUIDE

*(Unit tests, Spring context, @Mock vs @MockBean, real production usage)*

***

## 1️⃣ First: Understand the Two “Worlds”

This is the **most important concept**.  
Everything else makes sense only if this is clear.

### ✅ World 1: Mockito World (Pure Unit Tests)

*   No Spring context
*   No application startup
*   No DI container
*   Objects created by Mockito only

### ✅ World 2: Spring Boot World (Spring Context Tests)

*   Spring context is started
*   Beans are created by Spring
*   Dependency injection happens via Spring

➡️ **Mockito behaves VERY differently in each world**

***

## 2️⃣ When Mockito is Used WITHOUT Spring Boot

You’ve already learned this, but let’s anchor it.

### ✅ Pure Mockito Test (No Spring)

```java
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    UserRepository repository;

    @InjectMocks
    UserService userService;

    @Test
    void testUser() {
        when(repository.findById(1L))
            .thenReturn(new User(1L));

        User user = userService.getUser(1L);

        assertEquals(1L, user.getId());
    }
}
```

### ✅ Key Characteristics

*   ✅ No `@SpringBootTest`
*   ✅ Fast
*   ✅ True unit test
*   ✅ Mockito controls everything

📌 **This is the default and preferred way for unit tests**

***

## 3️⃣ Why Spring Boot Tests Are Different

Spring Boot tests introduce **the Spring Application Context**.

When Spring Boot starts:

*   It scans classes
*   Creates beans
*   Injects dependencies
*   Manages lifecycle

❗ Mockito **cannot automatically override Spring beans**  
This is where `@MockBean` comes in.

***

## 4️⃣ The Most Important Difference

## `@Mock` vs `@MockBean` (INTERVIEW FAVORITE)

***

### ✅ `@Mock` (Pure Mockito)

```java
@Mock
UserRepository repository;
```

*   Created by Mockito
*   Exists **only in test class**
*   Spring does NOT know about it
*   ❌ Not injected into Spring beans

✅ Use with:

```java
@ExtendWith(MockitoExtension.class)
```

***

### ✅ `@MockBean` (Spring Boot Integration)

```java
@MockBean
UserRepository repository;
```

*   Created by Mockito
*   **Registered as a Spring bean**
*   Replaces the real bean in Spring context

✅ Spring **injects this mock** everywhere the bean is used

📌 **This annotation connects Mockito ↔ Spring**

***

## 5️⃣ What EXACTLY Does `@MockBean` Do Internally?

This is critical to truly “get it”.

When Spring Boot sees:

```java
@MockBean
UserRepository repository;
```

Spring does the following:

1.  Creates a Mockito mock
2.  Registers it in the Spring Application Context
3.  Replaces the real `UserRepository` bean
4.  Injects the mock into:
    *   `@Service`
    *   `@Controller`
    *   `@Component`
    *   `@Autowired` fields

✅ One annotation, entire context substituted

***

## 6️⃣ Example: Spring Boot Service Test (Correct Way)

### Production Code

```java
@Service
class UserService {

    private final UserRepository repository;

    UserService(UserRepository repository) {
        this.repository = repository;
    }

    User getUser(Long id) {
        return repository.findById(id);
    }
}
```

***

### ❌ WRONG Test (Common mistake)

```java
@SpringBootTest
class UserServiceTest {

    @Mock
    UserRepository repository; // ❌ Spring doesn’t see this

    @Autowired
    UserService userService;
}
```

➡️ `UserService` still uses **real repository**

***

### ✅ CORRECT Test

```java
@SpringBootTest
class UserServiceTest {

    @MockBean
    UserRepository repository;

    @Autowired
    UserService userService;

    @Test
    void testGetUser() {
        when(repository.findById(1L))
            .thenReturn(new User(1L));

        User user = userService.getUser(1L);

        assertEquals(1L, user.getId());
        verify(repository).findById(1L);
    }
}
```

✅ Real Spring context  
✅ Mockito controls dependency  
✅ Proper integration test (service slice)

***

## 7️⃣ Controller Tests with Mockito + Spring Boot

***

### Production Controller

```java
@RestController
class UserController {

    private final UserService userService;

    UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/users/{id}")
    User getUser(@PathVariable Long id) {
        return userService.getUser(id);
    }
}
```

***

### ✅ Controller Test (NO full context)

```java
@WebMvcTest(UserController.class)
class UserControllerTest {

    @MockBean
    UserService userService;

    @Autowired
    MockMvc mockMvc;

    @Test
    void testEndpoint() throws Exception {
        when(userService.getUser(1L))
            .thenReturn(new User(1L));

        mockMvc.perform(get("/users/1"))
               .andExpect(status().isOk())
               .andExpect(jsonPath("$.id").value(1));
    }
}
```

### ✅ Why this is correct

*   ✅ No DB
*   ✅ No repository
*   ✅ Only MVC + controller layer
*   ✅ Mockito isolates service

📌 **This is called a “slice test”**

***

## 8️⃣ Mockito Annotations in Spring Boot Context

Let’s clearly define where each belongs.

| Annotation     | Used With    | Purpose                     |
| -------------- | ------------ | --------------------------- |
| `@Mock`        | Mockito only | Pure unit tests             |
| `@InjectMocks` | Mockito only | Inject mocks into object    |
| `@MockBean`    | Spring Boot  | Replace Spring bean         |
| `@SpyBean`     | Spring Boot  | Partial mock of Spring bean |
| `@Autowired`   | Spring       | Inject managed bean         |

***

## 9️⃣ `@SpyBean` — Partial Mocking in Spring Boot

### ✅ What it does

```java
@SpyBean
UserService userService;
```

*   Uses **real Spring bean**
*   Allows partial mocking
*   Dangerous if misused

***

### ✅ Example

```java
@SpyBean
UserService userService;

doReturn(new User(1L))
    .when(userService).getUser(1L);
```

⚠️ **Use sparingly**  
Senior rule:

> If you need `@SpyBean`, reconsider design.

***

## 🔟 Best Practice: Test Pyramid (CRITICAL)

### ✅ Recommended Distribution

    Integration Tests      (few)
    Spring Boot Tests      (some)
    Mockito Unit Tests     (many)

### ✅ Rule of Thumb

*   **Business logic** → Mockito + no Spring
*   **Wiring correctness** → Spring Boot + @MockBean
*   **Real flows** → Integration tests

***

## 1️⃣1️⃣ Common Interview Traps (IMPORTANT)

### ❌ Trap 1

> “@MockBean and @Mock are the same”

✅ **Wrong**

***

### ❌ Trap 2

> “Mockito can inject directly into Spring beans”

✅ **Wrong**

*   Only `@MockBean` interacts with Spring context

***

### ❌ Trap 3

> “Use SpringBootTest for everything”

✅ **Wrong**

*   It’s slow
*   Use slice tests or pure Mockito tests

***

## ✅ FINAL MENTAL MODEL (MEMORIZE THIS)

> **Use pure Mockito (`@Mock`, `@InjectMocks`) for fast unit tests without Spring.  
> Use `@MockBean` when Spring Boot creates your objects and you need Mockito to replace real beans in the application context.**

***

## ✅ One‑Line Interview Answer

> *Mockito is used in Spring Boot either as a standalone unit‑testing tool or integrated with the Spring context using `@MockBean` to replace real beans, allowing Spring‑managed components to be tested in isolation without invoking real infrastructure.*

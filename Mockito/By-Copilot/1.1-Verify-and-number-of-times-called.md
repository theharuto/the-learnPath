## 1️⃣ What is Verification in Mockito (Conceptual Meaning)

### ✅ Verification = checking interactions

Verification answers **questions about behavior**, not data:

*   Was this method called?
*   How many times was it called?
*   With what arguments?
*   In what order?
*   Was it *not* called?

Mockito verification checks **how your class collaborates with its dependencies**.

***

## 2️⃣ Minimal Example Setup (Context)

We’ll use this simple scenario throughout.

### Production code (class under test)

```java
class UserService {

    private final UserRepository repository;

    UserService(UserRepository repository) {
        this.repository = repository;
    }

    void register(Long id) {
        repository.findById(id);
        repository.save(new User(id));
    }
}
```

### Test setup

```java
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    UserRepository repository;

    @InjectMocks
    UserService userService;
}
```

***

## 3️⃣ Basic Verification: Was a Method Called?

### ✅ Syntax

```java
verify(repository).findById(1L);
```

### ✅ Meaning (very important)

This asserts:

> “During test execution, `findById(1L)` **must have been called at least once**.”

If it was **not called**, Mockito throws:

    Wanted but not invoked

***

### ✅ Full Example

```java
@Test
void shouldFindUserDuringRegistration() {
    userService.register(1L);

    verify(repository).findById(1L);
}
```

✅ Test passes if called  
❌ Test fails if not called

***

## 4️⃣ Verify Number of Times (Invocation Count)

This is where `times()`, `never()`, etc. come in.

***

### ✅ `times(n)` – exact number of calls

```java
verify(repository, times(1)).findById(1L);
```

Means:

> “This method must be called **exactly once**.”

***

### ✅ Example

```java
userService.register(1L);

verify(repository, times(1)).findById(1L);
verify(repository, times(1)).save(any(User.class));
```

If method is called:

*   0 times → fail
*   2 times → fail

***

## 5️⃣ `never()` – verify method was NOT called

### ✅ Syntax

```java
verify(repository, never()).delete(any());
```

### ✅ Meaning

> “This method **must not be called at all**.”

***

### ✅ Example

```java
userService.register(1L);

verify(repository, never()).delete(any());
```

✅ Passes if delete wasn’t called  
❌ Fails if delete was called even once

***

## 6️⃣ `atLeastOnce()` and `atLeast(n)`

### ✅ `atLeastOnce()`

```java
verify(repository, atLeastOnce()).findById(any());
```

Meaning:

> “This method must be called **one or more times**.”

***

### ✅ `atLeast(n)`

```java
verify(repository, atLeast(2)).findById(any());
```

Meaning:

> “Must be called **2 or more times**.”

***

### ✅ When to use this

*   Event-driven logic
*   Retry logic
*   Loops
*   Framework callbacks

***

## 7️⃣ Verifying Arguments (VERY IMPORTANT)

Mockito can verify **exact values** OR **use matchers**.

***

### ✅ Verify exact argument value

```java
verify(repository).findById(1L);
```

This checks:

*   Method name ✅
*   Argument value = `1L` ✅

***

### ❌ If argument is different

If code calls:

```java
findById(2L);
```

Mockito throws:

    Argument(s) are different!

***

## 8️⃣ Argument Matchers – partial matching

Matchers allow flexible verification.

### ✅ Common matchers

| Matcher       | Meaning      |
| ------------- | ------------ |
| `any()`       | Any object   |
| `anyLong()`   | Any long     |
| `anyString()` | Any string   |
| `eq(value)`   | Exact value  |
| `isNull()`    | Must be null |

***

### ✅ Example

```java
verify(repository).findById(anyLong());
```

Meaning:

> “Called with *some* long value (don’t care which).”

***

### ✅ Mixed matcher rule (INTERVIEW TRAP)

❌ This is **INVALID**:

```java
verify(repository).findById(anyLong(), 1);
```

✅ Must use matchers for **all arguments**:

```java
verify(repository).findById(anyLong(), eq(1));
```

***

## 9️⃣ Verify Both Count AND Argument Together

Very common in interviews.

```java
verify(repository, times(1)).findById(1L);
verify(repository, times(1)).save(any(User.class));
```

✅ Verifies:

*   Called once ✅
*   With correct argument ✅

***

## 🔟 Verifying No More Interactions

### ✅ `verifyNoMoreInteractions()`

```java
verifyNoMoreInteractions(repository);
```

Meaning:

> “No other methods should be called on this mock.”

### ✅ Use case

*   Ensuring side effects don’t happen
*   Tight unit tests

⚠️ Overuse can make tests brittle (senior insight)

***

## 1️⃣1️⃣ Verifying Order of Calls (Advanced but Important)

```java
InOrder inOrder = inOrder(repository);

inOrder.verify(repository).findById(1L);
inOrder.verify(repository).save(any());
```

Ensures:

> “Methods were called in this exact order.”

***

## 1️⃣2️⃣ Verification is Behavior Testing (Key Mental Model)

*   **Stubbing** → controls return values
*   **Verification** → checks behavior

| Question                | Use          |
| ----------------------- | ------------ |
| What value is returned? | Stubbing     |
| Was method called?      | Verification |
| How many times?         | Verification |
| With which args?        | Verification |

***

## 1️⃣3️⃣ Common Interview Traps (READ CAREFULLY)

### ❌ Trap 1: Verifying internal methods

Mockito verifies **interactions with dependencies**, not private methods.

***

### ❌ Trap 2: Verifying framework calls

Don’t verify Spring, Hibernate, or JDK internals.

***

### ❌ Trap 3: Over-verification

Too many `verify()` → brittle tests.

✅ Verify **important behavior only**.

***

## ✅ Final One‑Line Summary (INTERVIEW PERFECT)

> **Mockito verification allows you to assert how your class interacts with its dependencies by checking whether a method was called, how many times it was called, and with which arguments, ensuring correct behavior without using real external systems.**

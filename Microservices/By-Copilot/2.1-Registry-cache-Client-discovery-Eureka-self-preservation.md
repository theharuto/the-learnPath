# 1️⃣ Client‑Side Discovery — ultra‑deep explanation

## First: what is “the client” here?

### ✅ Client (important definition)

In microservices, **a “client” is NOT a browser**.

A **client** means:

> *Any microservice that wants to call another microservice*

Example:

    Order Service → calls → Payment Service

➡️ **Order Service is the client**
➡️ **Payment Service is the service being discovered**

***

## The core problem client‑side discovery solves

Imagine **Payment Service** has **3 running instances**:

| Instance  | IP       | Port |
| --------- | -------- | ---- |
| payment‑1 | 10.0.1.3 | 8080 |
| payment‑2 | 10.0.1.7 | 8080 |
| payment‑3 | 10.0.1.9 | 8080 |

These **change dynamically** because:

*   autoscaling
*   restarts
*   failures

❌ Order Service **cannot hard‑code** any of these.

***

## ✅ Client‑Side Discovery (what it literally means)

> **The client itself asks the service registry  
> downloads the list of available service instances  
> and decides which instance to call**

This is **not theory** — this is the **actual Eureka workflow**.

Baeldung and microservices.io both define Eureka as **client‑side discovery**.

***

## What does “client fetches the registry” ACTUALLY mean?

### Step‑by‑step (no skipping)

### ✅ Step 1: Client contacts Eureka Server

When **Order Service starts**:

It sends an HTTP call to Eureka Server:

    GET /eureka/apps

➡️ Eureka replies with **the entire registry**.

That registry looks conceptually like this:

    PAYMENT-SERVICE:
      - 10.0.1.3:8080
      - 10.0.1.7:8080
      - 10.0.1.9:8080

    INVENTORY-SERVICE:
      - 10.0.2.4:8090

📌 **Important**:  
This is **NOT just Payment Service** — it is the **full registry snapshot**.

***

### ✅ Step 2: Client stores it locally

Now Order Service keeps this data:

*   **In memory**
*   **Inside the JVM**
*   **No database**
*   **No disk**

This is the **local registry cache**.

✔️ Extremely fast  
✔️ No network call needed for lookups

***

### ✅ Step 3: Client makes a service call

Order Service wants to call Payment Service.

Instead of:

    http://10.0.1.7:8080/pay

It uses:

    http://PAYMENT-SERVICE/pay

Internally:

1.  Lookup `PAYMENT-SERVICE` **from local cache**
2.  Pick one instance (round‑robin by default)
3.  Call **that instance directly**

✅ This is client‑side **load balancing**

***

## ✅ Why this is called “client‑side” (important)

Because:

| Decision                 | Who does it |
| ------------------------ | ----------- |
| Which instances exist    | Client      |
| Which instance to call   | Client      |
| Load‑balancing algorithm | Client      |
| Fallback when one fails  | Client      |

➡️ Eureka Server **does NOT route traffic**  
➡️ Eureka Server **only stores information**

***

# 2️⃣ Registry Fetch & Local Cache — very detailed

You asked the **exact right question**:

> “cache locally means for how long? can we change the time?”

Yes — **100% configurable**.

***

## Where is the cache?

✅ In memory  
✅ Inside each microservice JVM  
✅ Managed by **Eureka Client library**

There is **NO shared cache** between services.

Each service has its **own copy**.

***

## How long is the cache valid?

### ✅ Default behavior (important numbers)

| Setting                      | Default        |
| ---------------------------- | -------------- |
| Registry fetch interval      | **30 seconds** |
| Heartbeat (renewal) interval | **30 seconds** |
| Eviction check (server)      | **60 seconds** |

So:

➡️ Every **30 seconds**, clients refresh the registry  
➡️ They do **not** call Eureka on every request

***

## ✅ Can we change registry fetch interval? YES

Spring property (official):

```properties
eureka.client.registry-fetch-interval-seconds=30
```

Example:

```properties
eureka.client.registry-fetch-interval-seconds=10
```

🔁 Now the client refreshes registry every **10 seconds**

***

## Why NOT fetch registry on every request?

Imagine:

*   200 services
*   each receiving 2,000 requests/sec

If **every request** queried Eureka:

❌ Eureka becomes a bottleneck  
❌ Huge latency  
❌ Registry outage = total system failure

***

## ✅ Cache avoids this problem

Flow becomes:

    Request → Local memory → Network call to service

NOT:

    Request → Eureka → Service

📌 **Eureka is removed from the hot path**

This is *the key design point*.

***

## “What if cache is stale?”

Good question — this is **expected and designed**.

*   Old instance might be called once
*   Client retries another instance
*   Next registry refresh removes bad instance

✅ Eventual consistency is acceptable  
✅ Full consistency would kill performance

***

# 3️⃣ Eureka Self‑Preservation Mode — deep senior explanation

This part **confuses many developers**, so let’s use a **real failure scenario**.

***

## First: heartbeats refresher

Every Eureka Client:

*   Sends heartbeat every **30 seconds**
*   Heartbeat = “I am still alive”

If heartbeats stop:
➡️ Eureka **normally removes the instance**

***

## ❌ The dangerous problem (network partition)

### Scenario: Network glitch (very realistic)

1.  100 Payment Service instances
2.  Network issue between Eureka and services
3.  **90 services suddenly fail to send heartbeat**
4.  They are actually **healthy**
5.  Network recovers after 40 seconds

### ❌ Without protection, Eureka would think:

> “90 services are dead — remove them!”

Now:

*   Registry instantly deletes 90 instances
*   Only 10 remain
*   Traffic explodes on remaining ones
*   Cascade failure happens

💥 **System melts down**

***

## ✅ Eureka Self‑Preservation Mode (what it really does)

> If Eureka detects **too many missed heartbeats at once**,  
> it assumes a **network problem**, not service death.

So Eureka says:

> “Something is wrong with the network — I will pause evictions.”

***

## How Eureka decides this (simple rules)

Eureka tracks:

*   Expected renewals
*   Actual renewals
*   Percentage received

If renewals fall below a threshold (default **85%**):

➡️ **Self‑preservation mode activates**
➡️ Eureka **stops removing instances**, even if heartbeats are missing

This behavior is **documented in Netflix Eureka & Spring Cloud docs**.

***

## ✅ What self‑preservation does NOT mean (important)

❌ It does NOT mean services are healthy  
❌ It does NOT mean requests are routed magically  
✅ It means **registry refuses mass deletion**

Clients still:

*   Use cached registry
*   Retry calls
*   Handle failures using timeouts and retries

***

## ✅ Why this is a senior‑level concept

Because it’s a **trade‑off**:

| Risk            | Without | With |
| --------------- | ------- | ---- |
| False deletion  | ✅       | ❌    |
| Stale entries   | ❌       | ✅    |
| Cascade failure | ❌       | ✅    |

Netflix learned this **from real production outages**.

***

## Can we disable self‑preservation?

Yes (but **dangerous**):

```properties
eureka.server.enable-self-preservation=false
```

⚠️ **Not recommended** for real systems.

***

# Final mental model (important)

### ✅ One‑line summary

> **Eureka clients periodically download the full service registry, cache it in memory, make service calls using that cache, and rely on self‑preservation to avoid catastrophic failures during network problems.**

> ❓ *When a service instance is down, who retries the request, how many times, using which component, and at what exact point in the Spring Cloud stack?*

***

# ✅ Short Answer (to set your expectation)

**Retry is NOT automatic by default.**  
When a service is down:

*   ❌ **Spring Cloud LoadBalancer does NOT retry by itself**
*   ❌ **ReactorLoadBalancer does NOT retry**
*   ✅ **Retries are handled by:**
    *   Spring Cloud LoadBalancer **+ Retry module** *(explicitly enabled)*
    *   OR Feign **Retryer** *(if configured)*
    *   OR Gateway **Retry filter** *(if configured)*
    *   OR Custom client logic

Now let’s prove and explain this **internally**.

***

# 1️⃣ What happens when a service is DOWN (baseline flow)

Let’s assume:

*   Service name: `USER-SERVICE`
*   Instances in Eureka:
        A → 10.0.1.1:8080 (DOWN)
        B → 10.0.1.2:8080 (UP)
        C → 10.0.1.3:8080 (UP)

Client makes a request:

```http
GET http://USER-SERVICE/users/1
```

***

## ✅ Internal Flow (WITHOUT retry enabled)

### Step 1: Service Discovery

*   **Eureka** provides list of instances (A, B, C)

✅ Eureka **does not know real‑time network failure**  
✅ It only knows last heartbeat status

***

### Step 2: Instance Selection

Component involved:

> **ReactorLoadBalancer (RoundRobinLoadBalancer)**

What it does:

*   Picks **ONE instance**
*   Example: chooses **Instance A**

✅ **No retry logic here**
✅ **Selection ≠ execution**

***

### Step 3: HTTP Call Execution

*   HTTP client (WebClient / Netty / Feign HTTP client) attempts call
*   Connection fails / timeout / connection refused

💥 **Exception thrown**

***

### Step 4: What happens next?

❌ LoadBalancer stops  
❌ ReactorLoadBalancer stops  
❌ No other instance tried

👉 **Request fails immediately**

***

## 🔴 Key Conclusion (VERY IMPORTANT)

> **Spring Cloud LoadBalancer ONLY SELECTS an instance.  
> It does not retry failed calls.**

This is **by design**.

***

# 2️⃣ Why LoadBalancer does NOT retry (design reason)

### Reason 1: Separation of concerns

Spring Cloud team **intentionally split responsibilities**:

| Component    | Responsibility |
| ------------ | -------------- |
| Eureka       | Who exists     |
| LoadBalancer | Who to try     |
| HTTP client  | Network IO     |
| Retry        | Failure policy |

✅ This keeps LoadBalancer **lightweight and deterministic**

***

### Reason 2: Retry is context‑dependent

Retry depends on:

*   HTTP method (GET vs POST)
*   Idempotency
*   Timeouts
*   Failure type

LoadBalancer **cannot decide this safely**.

***

# 3️⃣ So how DOES retry work in Spring Cloud?

There are **FOUR different retry layers**, depending on where the call is made.

***

# 4️⃣ Retry in Spring Cloud LoadBalancer (client‑side retry)

✅ **This is optional and must be explicitly enabled**

### How it works internally

Spring Cloud provides:

*   `RetryableLoadBalancerExchangeFilterFunction`

This acts as **a wrapper around LoadBalancer + HTTP client**.

***

### Internal retry algorithm (simplified)

1.  LoadBalancer selects instance A
2.  HTTP call fails
3.  Retry interceptor triggers
4.  LoadBalancer selects instance B
5.  HTTP call retried
6.  Stops when:
    *   Success
    *   Max attempts reached

✅ **This retry is instance‑aware**

***

### IMPORTANT: When is this enabled?

Only when:

*   `spring-cloud-starter-loadbalancer` **AND**
*   Spring Retry or Reactor Retry is on classpath
*   Retry is **explicitly configured**

🔴 **Without configuration → no retry**

***

# 5️⃣ Retry in Feign (Feign Retryer)

Feign has **its own retry mechanism**, totally separate.

***

### Feign Retry flow

When using **Feign + LoadBalancer**:

    Feign Retryer
       ↓
    LoadBalancer chooses instance
       ↓
    HTTP call

If failure:

*   Feign Retryer decides whether to retry
*   Calls LoadBalancer AGAIN
*   LoadBalancer returns next instance

✅ Feign **can retry across instances**
✅ But again: **only if enabled**

***

### Default Feign behavior

❌ Default Retryer = `Retryer.NEVER_RETRY`

So:

> **Feign does NOT retry by default**

***

# 6️⃣ Retry in Spring Cloud Gateway (Gateway Retry Filter)

Spring Cloud Gateway has an **explicit Retry filter**.

***

### Gateway retry flow (very important distinction)

    Client
     → Gateway
       → Retry Filter
           → LoadBalancer
               → Service instance

If instance fails:

*   Gateway retry filter triggers
*   LoadBalancer selects another instance
*   Request retried **from gateway level**

✅ This is **edge‑level retry**
✅ Very common in production

***

### Gateway can retry on:

*   Status codes (502, 503)
*   IO errors
*   Timeouts

But again:
❌ **Not automatic**
✅ Must be configured

***

# 7️⃣ ReactorLoadBalancer itself — VERY IMPORTANT CLARIFICATION

### Question you asked:

> *Does ReactorLoadBalancer handle retry?*

### ✅ Definitive answer:

❌ **NO**

`ReactorLoadBalancer`:

*   Chooses ONE `ServiceInstance`
*   Returns `Mono<ServiceInstance>`
*   Has **zero knowledge** of request outcome

It does **not know**:

*   Whether HTTP succeeded
*   Whether instance was slow
*   Whether timeout happened

✅ It is **stateless and passive**

***

# 8️⃣ What about health checks? Does LoadBalancer skip dead instances?

### Important truth (often misunderstood):

*   LoadBalancer **does NOT perform active health checks**
*   It trusts service discovery (Eureka)

If:

*   Eureka says instance exists → LoadBalancer may choose it
*   Even if instance is **actually dead**

This is why retries are critical.

***

# 9️⃣ Putting it all together (FULL INTERNAL FLOW)

### Case: Retry NOT enabled

    Client
     → LoadBalancer picks A
     → HTTP call fails
     → Exception returned
     → END

***

### Case: Retry enabled (Feign / Gateway / Retry Filter)

    Client
     → Retry wrapper
       → LoadBalancer picks A
          → HTTP fails
       → LoadBalancer picks B
          → HTTP success
     → Response returned

✅ Retry logic lives **OUTSIDE** LoadBalancer  
✅ LoadBalancer is reused for each attempt

***

# 🔥 FINAL MENTAL MODEL (READ CAREFULLY)

> **Spring Cloud LoadBalancer does NOT retry.  
> It only selects instances.  
> Retries are performed by higher‑level components (Feign, Gateway, Retry Interceptors), which re‑invoke the LoadBalancer for each retry attempt to get a new instance.**

***

| Layer                | Retry?       | Responsibility     |
| -------------------- | ------------ | ------------------ |
| Eureka               | ❌            | Instance registry  |
| ReactorLoadBalancer  | ❌            | Instance selection |
| HTTP Client          | ❌            | Network call       |

# 🎯 Microservices – Interview Questions & Answers

*(Spring Boot + Spring Cloud – Junior → Senior Level)*

***

## ✅ SECTION 1: Microservices Fundamentals

### Q1. What are microservices?

**Answer:**
Microservices is an architectural style where an application is built as a set of **small, independent services**, each:

*   Running in its **own process**
*   Focused on **one business capability**
*   **Independently deployable**
*   Communicating over **network protocols** (HTTP, messaging)

***

### Q2. What problems do microservices solve?

**Answer:**
Microservices solve:

*   Monolith scalability limits
*   Slow deployments
*   Tight coupling between modules
*   Whole application failure on partial failure
*   Team dependency bottlenecks

***

### Q3. What are the main challenges of microservices?

**Answer:**

*   Distributed system complexity
*   Network failures
*   Data consistency
*   Observability (logging, tracing)
*   Operational overhead (DevOps, infra)

***

### Q4. Should we always use microservices?

**Answer:**
No.  
Microservices are beneficial **only when scalability, team autonomy, and fault isolation are required**.  
For smaller systems, a **modular monolith** is usually better.

***

## ✅ SECTION 2: Monolith vs Microservices

### Q5. Compare monolith and microservices.

**Answer:**

| Aspect     | Monolith      | Microservices       |
| ---------- | ------------- | ------------------- |
| Deployment | Single unit   | Per service         |
| Scaling    | Whole app     | Individual services |
| Failure    | App‑wide      | Isolated            |
| Complexity | Low initially | High operationally  |

***

### Q6. Why is network failure a big issue in microservices?

**Answer:**
Because microservices communicate over the network, network calls can:

*   Time out
*   Fail partially
*   Be slow or unreliable

This does not happen in monoliths where method calls are in‑process.

***

## ✅ SECTION 3: Service Registry & Discovery (Eureka)

### Q7. Why do we need service discovery?

**Answer:**
Because in microservices:

*   Service instances start/stop dynamically
*   IPs and ports change
*   Hard‑coding URLs is unreliable

Service discovery allows **dynamic lookup of service locations**.

***

### Q8. What is a Service Registry?

**Answer:**
A Service Registry is a **central directory** that stores:

*   Service name
*   Instance IPs and ports
*   Health status

***

### Q9. Is Eureka server‑side or client‑side discovery?

**Answer:**
**Client‑side discovery**

Clients:

*   Fetch the registry
*   Cache it locally
*   Decide which instance to call

***

### Q10. How does Eureka work internally?

**Answer:**

1.  Service registers with Eureka on startup
2.  Sends heartbeats periodically
3.  Eureka maintains a registry
4.  Clients fetch and cache the registry
5.  Calls are made using cached data

***

### Q11. What is Eureka self‑preservation mode?

**Answer:**
A safety mechanism where Eureka **avoids removing instances aggressively** when many heartbeats are lost suddenly—assuming a **network issue** rather than real service failure.

***

### Q12. Does Eureka retry failed requests?

**Answer:**
❌ No.  
Eureka **only provides service information**. It does **not retry** requests.

***

## ✅ SECTION 4: Feign Client

### Q13. What is Feign?

**Answer:**
Feign is a **declarative REST client** in Spring Cloud that allows:

*   Writing HTTP clients as **Java interfaces**
*   Automatic request creation
*   Automatic JSON serialization/deserialization

***

### Q14. Why is Feign preferred over RestTemplate?

**Answer:**

*   Less boilerplate code
*   Built‑in integration with:
    *   Eureka
    *   LoadBalancer
*   Cleaner, declarative syntax
*   Easier testing and maintenance

***

### Q15. Does Feign know service IPs?

**Answer:**
❌ No.  
Feign uses **service names**, which are:

*   Resolved via Eureka
*   Load‑balanced by Spring Cloud LoadBalancer

***

### Q16. Is Feign blocking or non‑blocking?

**Answer:**
Feign is **blocking** and synchronous.

For reactive systems, `WebClient` is used instead.

***

### Q17. Does Feign retry requests automatically?

**Answer:**
❌ No (by default)

Retry requires:

*   Explicit Feign Retryer configuration
*   Or Spring Retry

***

## ✅ SECTION 5: API Gateway (Spring Cloud Gateway)

### Q18. What is an API Gateway?

**Answer:**
An API Gateway is a **single entry point** that:

*   Routes requests to backend services
*   Handles cross‑cutting concerns (auth, logging, rate limiting)

***

### Q19. Why do we need an API Gateway?

**Answer:**
Without a gateway:

*   Clients must know multiple service URLs
*   Security logic is duplicated
*   Versioning becomes difficult
*   Client‑service coupling increases

***

### Q20. What is Spring Cloud Gateway built on?

**Answer:**

*   Spring WebFlux
*   Project Reactor
*   Fully **non‑blocking reactive architecture**

***

### Q21. What are the core components of Spring Cloud Gateway?

**Answer:**

*   **Route** – destination service
*   **Predicate** – condition to match request
*   **Filter** – modify request or response

***

### Q22. What kinds of filters exist?

**Answer:**

*   Pre‑filters (before routing)
*   Post‑filters (after response)
*   Global filters (applied to all routes)
*   Route‑specific filters

***

### Q23. Does Gateway contain business logic?

**Answer:**
❌ No.  
Gateway should contain **ONLY cross‑cutting concerns**, never business logic.

***

## ✅ SECTION 6: Spring Cloud LoadBalancer

### Q24. What problem does load balancing solve?

**Answer:**
It distributes traffic across **multiple instances of the same service** to improve:

*   Scalability
*   Availability
*   Fault tolerance

***

### Q25. Is Spring Cloud LoadBalancer client‑side or server‑side?

**Answer:**
✅ Client‑side load balancing

The client chooses the service instance.

***

### Q26. What is the default algorithm in Spring Cloud LoadBalancer?

**Answer:**
**Round‑Robin**

Requests are distributed evenly across instances in sequence.

***

### Q27. What does `lb://SERVICE-NAME` mean?

**Answer:**

*   `lb` → use Spring Cloud LoadBalancer
*   `SERVICE-NAME` → resolved via Eureka

Used in:

*   Gateway
*   Feign
*   WebClient

***

### Q28. Does LoadBalancer know if a service instance is down?

**Answer:**
❌ No.

LoadBalancer trusts service discovery and does **not perform active health checks**.

***

## ✅ SECTION 7: Retry & Failure Handling (VERY IMPORTANT)

### Q29. Who handles retry when a service is down?

**Answer:**
Retry is handled by **higher‑level components**, not LoadBalancer.

| Component                        | Retry        |
| -------------------------------- | ------------ |
| Eureka                           | ❌            |
| ReactorLoadBalancer              | ❌            |
| Spring Cloud LoadBalancer (core) | ❌            |
| Feign Retryer                    | ✅ (optional) |
| Gateway Retry Filter             | ✅ (optional) |
| Spring Retry                     | ✅            |

***

### Q30. Does ReactorLoadBalancer retry requests?

**Answer:**
❌ No.

ReactorLoadBalancer:

*   Selects **one instance**
*   Is stateless
*   Has no knowledge of request outcome

***

### Q31. How does retry work internally when enabled?

**Answer:**

1.  Retry layer intercepts failure
2.  LoadBalancer is asked again
3.  New instance is selected
4.  Request is retried
5.  Stops on success or retry limit

***

### Q32. Is retry dangerous?

**Answer:**
Yes, if used without:

*   Timeouts
*   Circuit breakers

Retry alone can cause **retry storms** and **cascade failures**.

***

## ✅ FINAL INTERVIEW SUMMARY (MEMORIZE THIS)

> **Eureka tells who exists, LoadBalancer chooses who to try, Feign calls the service, Gateway controls entry traffic, and Retry is optional and must be explicitly configured.**

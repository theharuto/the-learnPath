# ✅ DEMONSTRATION: Spring Cloud Gateway with Filters

We will build **one API Gateway** that routes requests to **two backend services**:

*   `user-service`
*   `order-service`

We will use:

*   **Spring Cloud Gateway**
*   **Eureka for service discovery**
*   **Built‑in filters**
*   **A custom filter**

***

## 🧠 FIRST: What exactly are we building?

### CLIENT VIEW

    Client → http://localhost:8080/users/1
    Client → http://localhost:8080/orders/100

### INTERNAL VIEW

    Client
       ↓
    Spring Cloud Gateway
       ↓
    (User Service / Order Service)

✅ Client knows **ONLY the Gateway**

***

# 2️⃣ ROUTES CONFIGURATION (VERY IMPORTANT)

Spring Cloud Gateway routing is usually done in **`application.yml`**.

***

## 2.1 application.yml

```yaml
server:
  port: 8080

spring:
  application:
    name: api-gateway

  cloud:
    gateway:

      routes:

        - id: user-service-route
          uri: lb://USER-SERVICE
          predicates:
            - Path=/users/**
          filters:
            - name: RequestHeader
              args:
                name: X-Gateway
                value: SpringCloudGateway
            - name: AddResponseHeader
              args:
                name: X-Response-Time
                value: gateway-response

        - id: order-service-route
          uri: lb://ORDER-SERVICE
          predicates:
            - Path=/orders/**
          filters:
            - RewritePath=/orders/(?<segment>.*), /${segment}
```

Now we will **explain every single letter and word here**.

***

# 3️⃣ BREAKING DOWN A ROUTE (LETTER BY LETTER)

***

## ✅ `routes`

### Meaning:

> A **list of routing rules**

Each route:

*   Matches a request
*   Forwards it to a service

***

## ✅ `id: user-service-route`

### Meaning:

> A **unique identifier** for the route

Used for:

*   Logging
*   Debugging
*   Monitoring

🚫 Clients never see this.

***

## ✅ `uri: lb://USER-SERVICE`

### BREAK THIS DOWN CAREFULLY

#### `lb://`

*   `"lb"` = **LoadBalancer**
*   Tells Gateway to use **Spring Cloud LoadBalancer**

#### `USER-SERVICE`

*   Logical service name
*   Must match:
    ```properties
    spring.application.name=USER-SERVICE
    ```

✅ Gateway asks Eureka:

> “Give me all instances of USER‑SERVICE”

✅ Gateway picks one instance  
✅ Gateway forwards the request

***

## ✅ `predicates`

### Predicate = **Condition**

> *When this condition is TRUE, apply this route*

***

### ✅ `Path=/users/**`

Breakdown:

*   `/users/**` means:
    *   `/users/1`
    *   `/users/abc`
    *   `/users/profile/12`

✅ If path matches → route activated  
❌ If not → ignored

***

# 4️⃣ FILTERS (THIS IS THE CORE OF THE TOPIC)

***

## 4.1 WHAT IS A FILTER?

### ✅ Filter — definition

> A **filter modifies the request or response** as it flows through the Gateway.

There are **two moments** where filters operate:

1.  **Before request is forwarded** → *Pre Filter*
2.  **After response is received** → *Post Filter*

***

### REQUEST FLOW DIAGRAM

    Client Request
        ↓
    [ Pre Filters ]
        ↓
    Route → Service
        ↓
    [ Post Filters ]
        ↓
    Client Response

***

# 5️⃣ BUILT‑IN FILTERS (EXPLAINED IN DEPTH)

***

## ✅ Filter 1: `RequestHeader`

```yaml
- name: RequestHeader
  args:
    name: X-Gateway
    value: SpringCloudGateway
```

### What this does:

*   Adds a header to **outgoing request**

### Final request sent to service:

```http
GET /users/1
X-Gateway: SpringCloudGateway
```

✅ Use cases:

*   Identify traffic coming from Gateway
*   Debugging
*   Auditing
*   Security checks

***

## ✅ Filter 2: `AddResponseHeader`

```yaml
- name: AddResponseHeader
  args:
    name: X-Response-Time
    value: gateway-response
```

### What this does:

*   Adds header to **response returning to client**

### Client receives:

```http
X-Response-Time: gateway-response
```

✅ Used for:

*   Tracing
*   Performance monitoring
*   Debug headers

***

## ✅ Filter 3: `RewritePath`

```yaml
- RewritePath=/orders/(?<segment>.*), /${segment}
```

### BREAK THIS DOWN CAREFULLY

#### `/orders/(?<segment>.*)`

*   Regex match
*   Captures everything after `/orders/`

Example:

    /orders/123

#### `/123`

*   Forwarded path becomes `/123`

✅ Why is this used?
Microservice doesn’t need gateway-specific paths.

***

# 6️⃣ CUSTOM FILTER (VERY IMPORTANT FOR SENIOR LEVEL)

Now let’s create our **own filter**.

***

## 6.1 What is a Custom Gateway Filter?

> Code you write that runs **before or after routing**

Used for:

*   Authentication
*   Logging
*   Validation
*   Request blocking

***

## 6.2 Create a Custom Pre Filter

```java
@Component
public class LoggingGatewayFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                             GatewayFilterChain chain) {

        String path = exchange.getRequest().getURI().getPath();
        System.out.println("Incoming request path: " + path);

        return chain.filter(exchange);
    }
}
```

***

## 🧠 DEEP EXPLANATION

***

### ✅ `GlobalFilter`

Means:

> Apply to **ALL routes**

If you want route-specific → `GatewayFilter`

***

### ✅ `ServerWebExchange`

This object contains:

*   Request
*   Response
*   Headers
*   Path
*   Query params

Think of it as:

> **One complete HTTP transaction**

***

### ✅ `GatewayFilterChain`

Represents:

> The remaining filters + routing logic

If you **do not call**:

```java
chain.filter(exchange)
```

❌ Request stops  
❌ Client hangs

***

### ✅ `Mono<Void>`

Because:

*   Gateway is **reactive**
*   Response is asynchronous
*   No blocking allowed

***

# 7️⃣ PRE FILTER vs POST FILTER (VERY IMPORTANT)

### PRE FILTER

```java
// logic before calling service
return chain.filter(exchange);
```

### POST FILTER

```java
return chain.filter(exchange)
        .then(Mono.fromRunnable(() -> {
            System.out.println("Response sent");
        }));
```

***

# 8️⃣ WHY API GATEWAY FILTERS ARE POWERFUL

✅ Centralized security  
✅ Centralized rate limiting  
✅ Centralized logging  
✅ No duplicate code in services  
✅ Easy policy changes

***

# ✅ FINAL MENTAL MODEL (READ CAREFULLY)

> **Spring Cloud Gateway receives every request, evaluates predicates to choose a route, applies pre‑filters, forwards the request to a discovered service, applies post‑filters on the response, and returns the final response to the client — all in a non‑blocking reactive manner.**
